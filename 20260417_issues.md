## 1. 🔴 치명적 버그

### 1-1. `ui_gateway_node.py` — publisher 중복 생성

**위치:** `fuel_robot_pkg/ui_gateway_node.py:196`

```python
def publish_status(self, text: str):
    msg = String()
    msg.data = text
    self.status_pub = self.create_publisher(String, '/fueling/status', 10)  # ← BUG
    self.status_pub.publish(msg)
```

- 호출할 때마다 새 publisher 생성 → 메모리 누수
- 기존 구독자 QoS 상태 불안정해질 수 있음
- `__init__`에서 생성된 `self.status_pub` 를 그대로 사용해야 함

**수정 방향:** `self.status_pub = self.create_publisher(...)` 라인 제거

---

### 1-2. `_fueling_loop` — 완료 판정 경쟁 조건

**위치:** `web/app.py:74~112`

- Flask 쪽 `_fueling_loop` 이 주유량 카운팅 + 완료 판정 (step=4, status=success) 을 동시에 수행
- ROS2 쪽 `doosan_commander_node` 도 시퀀스 완료 시 `fueling_complete` publish → gateway 가 PATCH 로 status=success
- **두 곳에서 같은 task 의 status 를 success 로 덮어쓰는 경쟁**

**수정 방향:**
- Flask `_fueling_loop` 은 **주유량/금액 카운팅만** 수행
- `current_step=4` / `status=success` 판정은 **ROS2 시퀀스 완료 시에만** 수행
- `_fueling_loop` 내부에서 target 도달 시 카운팅 중지만 하고 status 는 건드리지 않음

## 2. 🟡 예외 처리 보강

### 2-1. 3회 재시도 + abort 로직 (commander)

**위치:** `fuel_robot_pkg/doosan_commander_node.py`

**추가할 상태:**
```python
self.retry_count = 0
self.MAX_RETRIES = 3
self.current_step_snapshot = None  # 재시도용 현재 액션 보관
```

**로직:**
```
_step_done_cb(future, done_cmd):
    success = resp.success (or False on exception)
    
    if success:
        retry_count = 0
        completed_steps += 1
        publish_status('fueling_progress:N/T')
        _run_next_step()
    else:
        retry_count += 1
        if retry_count < MAX_RETRIES:
            publish_status('fueling_retry:N/3')
            re-execute current_step_snapshot
        else:
            publish_status('fueling_aborted')
            step_queue.clear()
            is_running = False
            execute_go_home()  # 자동 홈 복귀
```

### 2-2. E-STOP 후 task_manager FSM 리셋 체인

**위치:** `fuel_robot_pkg/fueling_task_manager_node.py`

- 현재 `robot_cmd_callback` 에서 'estop' 받으면 `reset_state()` 호출은 되지만
- 사용자가 **웹에서 바로 E-STOP** 누르면 gateway → commander 로 직접 감
- task_manager 가 E-STOP 을 "본인 FSM 초기화 신호"로 인식하는지 재확인 필요

**수정 방향:**
- gateway 의 `/fueling/estop` 핸들러가 `/fueling/robot_cmd` 에 'estop' publish → task_manager 도 같은 토픽 구독 중이므로 reset 됨 ✓ (이미 되어 있음)
- 단 `publish_done(False)` 가 gateway → Flask 까지 전파되는지 경로 확인 필요

### 2-3. 비전/Gateway 사전 체크

**위치:** `web/app.py:/api/task/start`

현재:
```python
det = get_latest_detection(station_id)
if not det or det.get("robot_x") is None:
    insert_log("ERROR", ...)
    return jsonify({"ok": True, "task_id": task_id, "warn": "..."}), 201
```

**문제:**
- 비전 없어도 task 생성하고 201 반환 → DB 에 고아 task 남음
- Gateway 연결 실패해도 201 반환 → 사용자는 성공인 줄 알고 대기

**수정 방향:**
- 비전 det 없으면 task 생성 **안 함**, 400 반환 ("비전 인식 대기 중")
- Gateway POST 실패 시 방금 생성한 task 를 `status='fail'` 으로 즉시 업데이트 후 503 반환

## 5. 📝 로그 체계 개선

### 5-1. Vision 측 로그 전송 추가

**위치:** `vision/post_vision.py`

- 현재 Flask 로 전송하는 건 detect 데이터 + frame 뿐
- 다음 이벤트를 `POST /api/logs` 로 전송:
  - **ERROR** `카메라 프레임 드랍` (depth/color frame 연속 null)
  - **ERROR** `RealSense 파이프라인 재시작`
  - **WARN** `신뢰도 하락 (label=X, conf=Y)` — conf < 0.70 인 경우 (rate-limit 적용: 5초마다 1회)
  - **INFO** `카메라 초기화 완료` (기동 시 1회)

**구현:**
```python
def log_to_flask(level, message):
    try:
        requests.post(f"{API_URL}/api/logs", json={
            'level': level, 'message': message,
            'source': 'vision', 'station_id': STATION_ID,
        }, timeout=0.3)
    except Exception:
        pass
```
